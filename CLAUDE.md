## Supabase Gotchas
- **RLS-by-default for new public tables (2026-04-27 lesson).** Every `CREATE TABLE public.X` migration MUST be followed by `ALTER TABLE public.X ENABLE ROW LEVEL SECURITY`, even for service-role-only tables (use no policies — service role bypasses RLS). Incident: `bot_settings` (added 2026-04-19 in post-ban rush) was missed → Supabase Security Advisor flagged it CRITICAL because anon key (embedded in React bundle) could flip kill switch flags via REST API. Fixed via migration `2026_04_27_enable_rls_bot_settings.sql`. Audit query: `SELECT tablename, rowsecurity FROM pg_tables WHERE schemaname='public' AND rowsecurity=false;`
- **Custom auth domain (`auth.sheli.ai`) is auth-only.** REST API client config in `src/lib/supabase.js` must keep using `wzwwtghtnkapdwlgnrxr.supabase.co` as `supabaseUrl` in the `createClient` call. Custom domain ONLY affects `/auth/v1/*` endpoints (OAuth callbacks, sessions). REST, Realtime, Storage, and Edge Functions all continue resolving via the project URL. Don't change `createClient` args when adding a custom domain.
- **RLS blocks everything when auth.uid() is NULL** — Supabase client with publishable key + auth session sends JWT. If JWT is stale/expired, auth.uid() returns NULL and all RLS policies fail silently.
- **Clock skew warning** (`Session as retrieved from URL was issued in the future`) — indicates JWT timestamp mismatch. Can cause auth.uid() to be NULL. Fix: clear localStorage and re-authenticate.
- **RLS tightened for launch (2026-04-08)** — all core tables use `is_household_member(household_id)`. See security audit: `docs/plans/2026-04-08-security-audit-design.md`.
- **Use JWT anon key, NOT publishable key** — `sb_publishable_...` format causes 406 errors on raw REST calls (PostgREST rejects non-JWT tokens). Always use the legacy `eyJhbG...` anon JWT. Both are available in the project.
- **Edge Functions use service_role key** (bypasses RLS). Web app uses anon JWT key (goes through RLS).
- **Realtime must be explicitly enabled** per table: `ALTER PUBLICATION supabase_realtime ADD TABLE public.tablename;`
- **`household_members` has `USING (true)` fallback policy** — needed because clock skew made auth-based policies unreliable

## React / Boot Flow Gotchas
- **`<input>` inline styles MUST set both `background` AND `color` (2026-04-27).** Browser user-agent stylesheet defaults `color: black` on inputs; if you set `background: var(--white)` (which swaps to dark `#1C2424` in dark mode) without also setting `color: var(--dark)` (which swaps to light `#E4ECEC`), input text is black-on-dark = invisible. The class-based `.field-input` in `src/styles/app.css` does this correctly; **inline-style fallbacks elsewhere are the trap.** Hit in `App.jsx` picker fallback line ~885 — the "Couldn't save your name" picker had unreadable input text. Also add `outline: "none"` for focus styling consistency.
- **localStorage `sheli-hhid` carryover causes "stuck on welcome" loops during testing.** Deleting a household server-side leaves the React app trying to load the cached household_id from localStorage → "Welcome, Cohen / Who are you?" loop with INSERT failures. When debugging account state, always test in incognito or run `localStorage.clear(); location.reload()` in Console. Sign-out also doesn't always fully clear — the boot useEffect can re-hydrate from a stale token before the logout completes.
- **Setup atomicity gap (orphan-with-key bug).** When `Setup.jsx` creates the household but the founder `household_members` insert fails (RLS or network), the household exists with `created_by` pointer but zero member rows → `detectHousehold()` Method 2 (`households_v2.created_by = userId`) finds it on next sign-in → routes to JoinOrCreate picker → picker INSERT also fails RLS (creator isn't a member yet) → infinite loop. Manual cleanup: `DELETE FROM households_v2 WHERE id=X AND created_by=Y` (CASCADEs cleanly when zero members). Long-term fix: wrap Setup in transaction or RPC, OR add RLS policy bypass for `created_by = auth.uid()`.
- **English `hhPlaceholder` should never contain a real-looking name.** Original `"The Cohen Home"` (en.js) caused users to type "Cohen" verbatim, creating ghost households named "Cohen" forever. Multi-suggestion list pattern (matching Hebrew `"משפחת ישראלי, המשפחה שלנו, כל שם שתבחרו"`) signals "this is just an example" much better. ## WhatsApp Bot Gotchas
- **Whapi "Persistent webhook" MUST stay ON** (toggle in panel.whapi.cloud → channel → Settings → Webhooks). Default is OFF. With it OFF, any transient 5xx from the Edge Function (e.g. during a Dashboard paste deploy) drops the inbound permanently — no retry. Confirmed live 2026-04-20 when a test message arrived at the bot phone exactly when v165 was deploying and was never forwarded to the webhook. Turn ON after any channel re-pair.
- **Webhook dedup window must stay narrow (≤5 min)**. `Deno.serve` has a dedup check at the top of `try` block that skips any message whose `whatsapp_message_id` matches an existing `whatsapp_messages` row. The predicate MUST include `.gte("created_at", now - 5 min)` — it exists only to suppress `sendAndLog → Whapi-echo` bounces which arrive within seconds. Before 2026-04-20 Tier A fix, the check matched all-time → history-sync replays after Whapi reconnect were silently eaten when they collided with `backlog_imported_user` rows from the 2026-04-18 recovery backfill (97 users × ~3 landmine IDs). If you ever think "let me widen this back" — re-read `docs/plans/2026-04-20-webhook-gap-fix-plan.md` first.
- **Webhook exit instrumentation: `[Webhook:EXIT:<code>]` + Tier B `logMessage` at every `handleDirectMessage` silent exit**. Grep Dashboard logs by `[Webhook:EXIT:` to reconstruct drop inventory. Four handler exits that historically left no DB trace (`dm_waitlist_new`, `dm_waitlist_reping`, `dm_rename`, `dm_joined_safety`) now always insert a row with `household_id='unknown'` sentinel. Helper is `logExit()` defined just above `Deno.serve`. Never remove these — they're the only observability for the waitlist-redirect path.
- **`onboarding_conversations.state` has a CHECK constraint** — any new state value added in code MUST also be added to the DDL constraint, otherwise `.insert()` fails **silently** (the code does `await supabase.from(...).insert(...)` without checking the error return). 2026-04-20 landmine: `'waitlist_redirected'` shipped in code 2026-04-19 but wasn't in the CHECK list → every waitlist-redirect insert failed invisibly → users looped on full `waitlistBody` text forever because `convo` stayed null. Fixed in migration `2026_04_20_fix_onboarding_state_check.sql` (added `waitlist_redirected` + `welcome_queued`). Current allowed states: `welcomed, chatting, sleeping, nudging, invited, joined, personal, dormant, welcome, trying, waiting, onboarded, active, waitlist_redirected, welcome_queued`. When you add a new state, `git grep 'state.*=.*"<new>"' supabase/functions/` to catch code usage AND update the constraint in the same migration.
- **Recurring reminders — first-class via parent+children (2026-04-20 Fix 4).** `reminder_queue` has `recurrence JSONB` (shape `{days:[0..6], time:"HH:MM"}`, 0=Sunday) + `recurrence_parent_id UUID` self-FK. Parents carry `recurrence` + `sent=true` (drain ignores them); the PG function `materialize_recurring_reminders()` runs daily at 01:00 UTC via pg_cron `materialize_recurring_reminders_daily` and inserts child rows for the next 7 days with `recurrence_parent_id` set. Children look like regular one-shot reminders to the drain — inherit `il_window_open_for_chat` gating. Two emission paths, both ship parent rows + trigger immediate materialize: **group** via `<!--RECURRING_REMINDER:{...}-->` HTML comment → `rescueRemindersAndStrip`; **1:1** via `{"type":"recurring_reminder","text":"...","days":[...],"time":"HH:MM"}` in the ACTIONS JSON → `execute1on1Actions` case handler. Rotations where different people are responsible on different days = TWO parent rows (one per person). Migration: `2026_04_20_recurring_reminders.sql`. Haiku classifier does NOT yet emit a dedicated `add_recurring_reminder` intent — it falls through to Sonnet; a future update can add the intent for faster routing.
- **Quiet hours no longer block user-scheduled reminders (2026-04-20).** Previous `fire_due_reminders_inner` had a top-level `IF is_quiet_hours_il() THEN RETURN 0` that suppressed EVERYTHING during 22:00-07:00 + Fri 15:00 – Sat 19:00 — including reminders the user explicitly scheduled for quiet hours (e.g. "תזכירי לי בשבת בבוקר ב-9"). Removed in migration `2026_04_20_quiet_hours_user_override.sql`: `fire_due_reminders_inner` v3 has no quiet-hours gate. Anti-spam remains gated per-row by `il_window_open_for_chat` (Meta 24h customer-care window) — the real anti-spam line. Children of a RECURRING_REMINDER inherit the same behavior: if the user picked a time that falls in quiet hours (e.g. Shabbat 19:00 shower reminder), it fires — the user chose that time. Out-of-window rows (recipient hasn't messaged in 24h) still get marked `sent=true + metadata.note='window_closed'` and are not retried.
- **Synthetic household auto-merge on real-group arrival (2026-04-20 Fix 3).** `import_chat_exports.py` creates synthetic households with `metadata = {synthetic_group:true, awaiting_real_jid:true}` for every .txt export. When the real WhatsApp group later adds Sheli, `handleBotAddedToGroup` at `index.inlined.ts:~4730` now scans for synthetic siblings by name (exact match OR `{name} (recovered)`) and, if found, promotes the oldest one to real + folds others into it (moves `whatsapp_messages`, `tasks`, `shopping_items`, `events`, `reminder_queue`) before deleting the extras. No new household is created — message history + any imported items merge into the live group. Metadata gains `merged_from_synthetic_at` + `merged_synthetic_count`. - **NO PHANTOM REMINDERS / NO FALSE-APOLOGY / REPLY-METADATA-DAY-CONSISTENCY rules (2026-04-26 Bilin + Globerman).** Three ABSOLUTE RULE blocks added to `SHARED_GROUNDING_RULES` + mirrored as 13c/13d/13e in `ONBOARDING_1ON1_PROMPT`. ## WhatsApp Bot — Two-Stage Pipeline (deployed 2026-04-03)

### Architecture
```
Message → Pre-filter (skip media/bot msgs) → Haiku Classifier ($0.0003) → Route:
  intent=ignore + conf≥0.70     → STOP (no Sonnet, 80% of messages)
  actionable + conf≥0.70        → Execute actions + Sonnet reply ($0.01)
  conf 0.50-0.69                → Escalate to full Sonnet classification (fallback)
  conf<0.50                     → Treat as ignore, log for review
```

- **Accuracy:** 91.7% on 120 test cases (target was 85%). `ignore` 100%, `add_event` 100%, `question` 100%.

## RTL / Hebrew Design Rules
- `dir="rtl"` on parent flips flexbox automatically — most layouts "just work"
- **Hebrew CTAs: always gender-free plural** — "המשיכו" (not "המשך"), "הירשמו" (not "הירשם"), "התחברו" (not "התחבר"). Masculine plural = universal form in modern Hebrew UX.
- **Hebrew singular pronouns that ARE gender-neutral:** לך / אותך / בשבילך are written identically for masc + fem in unpointed Hebrew. Prefer them over plural (לכם/אתכם/בשבילכם) when the copy addresses an individual in a question or personal statement — more intimate without introducing gender. Rule of thumb: switch to singular if it only touches pronouns ("מה הכי מעניין אותך?"); stay plural if it would force a gendered verb ("הירשם/י" → use "הירשמו"). Added 2026-04-19 after waitlist copy.
- Arrows: forward = ← in RTL, → in LTR. Back = → in RTL, ← in LTR.
- `letter-spacing: 0` on Hebrew text (uppercase letter-spacing breaks Hebrew)
- Font: Heebo for Hebrew, Nunito for English. Cormorant Garamond removed (too "luxury editorial" for a family app).
- **Hebrew copy:** use "מטלות" not "משימות" (tasks — משימות sounds military). Sheli speaks feminine first-person ("הוספתי" not "הוסף").
- **Copywriting skill:** `.claude/skills/copywriting.md` — guidelines for taglines, CTAs, FAQ, bot messages, social media (HE+EN).
- WhatsApp mock on welcome screen: force `direction: ltr` on bubble layout (WhatsApp always shows your msgs on right), inner text gets `direction: rtl` for Hebrew
- CSS logical properties: use `padding-inline-end` not `padding-right`, `inset-inline-end` not `right`
- **Arrows in WhatsApp messages:** ASCII arrows (`->`, `<-`, `→`, `←`) render unpredictably in RTL. Use numbered steps instead.
- **Hebrew single-letter prefix + hyphen + URL / Latin noun is BANNED** (2026-04-23): `ב-sheli.ai`, `ל-Google Calendar`, `מ-WhatsApp`, `כ-iCount` all render broken in RTL and can break URL auto-linking in WhatsApp. Worse — the pattern corrupts Sonnet's own output (2026-04-23 live: Sheli typed `בכ-sheli.ai`, a concat artifact of the `ב` letter stuttering at the prefix). Use one of three fixes: (1) URL on its own line with no prefix (`הכל פה:\nsheli.ai`), (2) Hebrew CARRIER WORD instead of a single letter (`באתר sheli.ai`, `ביומן Google`, `באפליקציית WhatsApp`), (3) reword so the URL / proper noun starts the clause (`sheli.ai מרכז את הכל`). Exception: `ה-` (definite article) before a Latin name is idiomatic and allowed (`ה-iPhone שלי`). Only `ב / ל / מ / כ / ש` + hyphen are banned. Rule also lives in `SHARED_HEBREW_GRAMMAR` so it's applied in both 1:1 and group Sonnet replies.
- **Section titles in Hebrew:** Use Heebo font (not Cormorant Garamond). Cormorant Garamond is ONLY for the English "Sheli" wordmark.

### Positioning implication

Meta's "AI Providers" discretion killed a $12M/5.5M-user product despite Cloud API compliance. Sheli must be framed as **"family task coordination utility"** not "AI assistant" — in Business verification, landing page copy, template category choice, FB post framing. Current tagline "העוזרת החכמה שלכם" is policy-risky; Boti's "השליח שלך" (your messenger) is safer positioning. Keep AI personality in-chat (delight), lead externally with shopping/reminders/family coordination (utility).

### The three survival architectures (2026)

1. **Cloud API + Utility templates** (Any.do, Reminder Bot) — paid, regulated, templates, punctual. ~$0.15/user/month.
2. **Device-side automation** (Wareminders, SKEDit) — no bot phone, user's own phone sends.
3. **Unofficial linked-device API (Whapi-class) + 100% reactive** (Boti) — sub-2% ban rate reactive-only, 15-30% if proactive. Means no punctual reminders.

Sheli currently attempts #3 with proactive reminders → bans. Decision pending.
