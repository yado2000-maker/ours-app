# WhatsApp OTP Authentication — Design Doc

**Date:** 2026-04-08
**Status:** Approved
**Author:** Claude (PMO/CTO) + Yaron

## Problem

Sheli's phone OTP auth UI is built (`AuthScreen.jsx`) but has no SMS provider configured. Twilio SMS to Israel costs $0.26-0.31/verification — expensive for a freemium app. Need a cost-effective OTP delivery mechanism.

## Decision

**WhatsApp OTP primary + Vonage SMS fallback**, delivered via Supabase Send SMS Hook.

- **Phase 1:** Whapi.Cloud (already paid $12/mo flat, unlimited messages) for WhatsApp delivery
- **Phase 2:** Meta Cloud API migration when scaling (adds auto-fill, $0.0053/msg)
- **SMS fallback:** Vonage ($0.057/verification, success-only billing)

### Why not Twilio?

| Provider | Cost/OTP to Israel | Monthly @ 1K OTPs |
|----------|-------------------|-------------------|
| Twilio Verify + SMS | $0.31 | $310 |
| Twilio SMS only | $0.26 | $260 |
| Vonage Verify | $0.057 | $57 |
| **Whapi.Cloud WhatsApp** | **$0 (flat $12/mo)** | **$12** |
| Meta Cloud API WhatsApp | $0.0053 | $5.30 |

WhatsApp covers ~95% of Israeli adults. SMS fallback handles the rest.

## Architecture

```
Phone Auth Flow:
═══════════════
AuthScreen (existing, no changes)
    → supabase.auth.signInWithOtp({ phone })
    → Supabase generates 6-digit OTP (10min TTL)
    → Supabase calls Send SMS Hook

Send SMS Hook (new Edge Function: otp-sender):
══════════════════════════════════════════════
Receives { phone, otp } from Supabase Auth Hook
    → Normalize phone (0XX → 972XX)
    → Whapi.Cloud GET /contacts/check → has WhatsApp?
        ├─ YES → POST /messages/text (DM with OTP)
        │        Log: whatsapp_messages (classification: "otp_sent")
        │        Return 200
        └─ NO  → Vonage SMS API (POST /sms/json)
                 Log: channel "sms"
                 Return 200

Post-Auth Bridge (new, for users without household only):
═════════════════════════════════════════════════════════
After OTP verified → user authenticated → web app boots
    → detectHousehold finds no household
    → Call otp-bridge endpoint or client-side trigger
    → WhatsApp DM: "התחברת בהצלחה! רוצה להוסיף אותי לקבוצת המשפחה?"
```

## Edge Function: `otp-sender`

### Env Vars

| Variable | Source | Purpose |
|----------|--------|---------|
| `WHAPI_TOKEN` | Existing | Whapi.Cloud bearer token |
| `WHAPI_URL` | Existing | Whapi.Cloud base URL |
| `VONAGE_API_KEY` | New (Vonage signup) | SMS fallback |
| `VONAGE_API_SECRET` | New (Vonage signup) | SMS fallback |
| `VONAGE_SENDER` | New | "Sheli" or Vonage number |

### Supabase Hook Payload

Supabase Send SMS Hook sends:
```json
{
  "user": { "phone": "+972501234567" },
  "sms": { "otp": "482019" }
}
```

Edge Function must return HTTP 200 with JSON body for Supabase to consider delivery successful.

### Logic (pseudocode)

```typescript
// 1. Parse hook payload
const { user, sms } = await req.json()
const phone = normalizePhone(user.phone) // → "972501234567"
const otp = sms.otp

// 2. Check WhatsApp presence
const hasWhatsApp = await whapiCheckContact(phone)

// 3. Deliver OTP
if (hasWhatsApp) {
  await whapiSendText(`${phone}@s.whatsapp.net`, otpMessage(otp, lang))
  await logOtpDelivery(phone, "whatsapp")
} else {
  await vonageSendSms(phone, smsMessage(otp))
  await logOtpDelivery(phone, "sms")
}

// 4. Return success to Supabase
return new Response(JSON.stringify({ success: true }), { status: 200 })
```

## Supabase Configuration

1. **Auth → Phone Auth → Enable** phone sign-in
2. **Auth → Hooks → Send SMS** → enable, point to `otp-sender` Edge Function URL
3. No Twilio/Vonage credentials in Supabase dashboard (handled by Edge Function)
4. OTP settings: 6 digits, 10 minute expiry (Supabase defaults)

## Frontend Changes

**AuthScreen.jsx:** No changes needed. Existing `handlePhoneSend` and `handlePhoneVerify` work as-is.

**App.jsx (boot flow):** Small addition — after phone auth, if no household detected, trigger post-auth bridge message via a lightweight API call to `otp-bridge` Edge Function.

## Security

- OTP generated by Supabase (cryptographically random, 6 digits, 10min TTL)
- Supabase handles attempt counting and lockout (5 attempts max)
- WhatsApp delivery is end-to-end encrypted
- Edge Function uses `SUPABASE_SERVICE_ROLE_KEY` (bypasses RLS)
- Phone normalization validates format before sending
- Rate limiting: Supabase built-in (prevents OTP spam)

## Limitations (Phase 1)

- **No auto-fill:** Whapi.Cloud sends regular messages, not Meta authentication templates. Users copy-paste the 6-digit code.
- **Contact check latency:** Whapi `/contacts/check` adds ~1-2s. Acceptable for auth.
- **Same bot number:** OTP comes from Sheli's bot number (+972 55-517-5553). This is intentional — creates a 1:1 thread that serves as an onboarding bridge.

## Phase 2: Meta Cloud API Migration

When to migrate: volume exceeds ~2,300 OTPs/mo, OR want auto-fill, OR migrating bot to Meta Cloud API.

### What changes

| Component | Phase 1 (Whapi) | Phase 2 (Meta Cloud API) |
|-----------|-----------------|--------------------------|
| Contact check | `GET /contacts/check` | Not needed (Meta handles) |
| Send message | `POST /messages/text` | `POST /{phone-id}/messages` (auth template) |
| Auth header | `Bearer {WHAPI_TOKEN}` | `Bearer {META_ACCESS_TOKEN}` |
| Message format | Free text | Pre-approved template with `{{otp}}` param |
| Auto-fill | No | Yes (Android native) |
| Cost/OTP | $0 (flat $12/mo) | $0.0053/msg |

### What stays the same

- Supabase Send SMS Hook architecture
- Edge Function receives `{ phone, otp }` 
- Vonage SMS fallback
- Frontend code
- Post-auth bridge logic
- Logging

### Migration steps

1. Apply for Meta Business verification
2. Set up WhatsApp Business API (Cloud API)
3. Submit authentication template for approval
4. Update `otp-sender` Edge Function: swap Whapi send → Meta Cloud API send (~20 lines)
5. Add `META_ACCESS_TOKEN`, `META_PHONE_ID` to Edge Function secrets
6. Test with verified numbers → go live

## Interaction with Existing WhatsApp Bot

- OTP messages are 1:1 DMs (never appear in family groups)
- Bot's `handleDirectMessage` won't interfere — OTP is outbound only (bot sends, user doesn't reply in WhatsApp)
- If user later messages Sheli in the same 1:1 thread, onboarding handler works normally
- `whatsapp_messages` log distinguishes OTP (`classification: "otp_sent"`) from regular messages
